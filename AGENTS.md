# AGENTS.md

## Purpose
- This repository is used for self-education in practical data engineering.
- Optimise for learning clarity, reproducibility, and explicit reasoning over "quick hacks."
- Prefer changes that make debugging and recovery easier in local environments.

## Language Preferences
- Prefer Kotlin whenever a JVM path is available (Spark notebooks, examples, utilities).
- Scala is acceptable when it is the cleaner JVM-native option.
- Use Python only when required by tooling/ecosystem constraints or when a chapter already depends on Python scripts.
- When proposing options, present Kotlin first, Scala second, Python fallback.

## English Variant
- Use Australian English spelling and usage in documentation, learner notes, code reviews, technical explanations, and agent responses.
- Prefer forms such as `optimise`, `behaviour`, `organise`, and `catalogue` where appropriate.

## Development Environment Expectations
- Assume chapter-level local development via Docker Compose.
- Keep startup/bootstrap flows idempotent by default.
- Put destructive resets behind explicit opt-in flags (for example, `RESET_WAREHOUSE_ON_STARTUP=true`).
- Keep catalog state and object storage state aligned; avoid partial resets that create stale metadata pointers.
- Prefer resilient downloads and initialisation steps (retryable downloads, atomic writes, local fallback data where possible).
- Validate changes with practical container lifecycle checks (`docker compose build`, `docker compose up`, relevant service health/log checks).

## Coding Style
- Favour defensive shell scripting patterns for init/bootstrap tasks:
  - `set -euo pipefail`
  - explicit variable names and quoting
  - clear skip/copy/fallback branches with informative log lines
- Keep behaviour deterministic for notebooks and demos where possible.
- Align cross-service config values explicitly (for example, warehouse roots across Spark and REST catalog).
- Keep patches scoped to the chapter and problem being solved; avoid unrelated refactors.

## Writing style

When writing explanations, notes, docs, PR descriptions, commit messages, or user-facing summaries:

- Prefer clear prose over dense bullet lists.
- Default to short paragraphs with smooth transitions between ideas.
- Start with a direct answer or framing sentence, then explain the why.
- Expand technical points enough for a careful learner, not just an expert skimming.
- Use bullets only when sequence, checklists, or comparisons genuinely benefit from list format.
- Avoid fragmented one-line bullets when a paragraph would read more naturally.
- Prefer warm, polished, teacherly writing over terse, robotic writing.
- Explain key concepts before jumping into fixes or commands.
- When refining existing writing, preserve the author's meaning but improve flow, clarity, precision, and completeness.
- Add helpful context, edge cases, and mental models when they improve understanding.
- Keep technical writing readable: define terms, reduce jargon density, and avoid abrupt topic jumps.
- Use headings to create a narrative structure, not just a dump of sections.
- When appropriate, include: context, diagnosis, root cause, fix, and takeaway.
- Prefer “show understanding” over “compress everything.”
- Do not default to ultra-short answers unless the user explicitly asks for brevity.

### Rewrite behaviour

When asked to review, refine, or enrich a piece of writing:

- First preserve the core intent and factual meaning.
- Then improve structure, flow, transitions, and completeness.
- Fill in missing rationale, not just grammar.
- Make the writing feel deliberate and finished, not just corrected.
- Where useful, strengthen with conceptual framing, examples, or a better mental model.
- Avoid turning everything into lists unless the source is clearly meant to be a checklist.

## Tone

- Sound like a strong technical mentor, not like sparse autogenerated notes.
- Be warm, confident, and explanatory.
- Optimise for understanding by a motivated learner, not for minimal token count.
- Avoid robotic phrasing.
- Avoid excessive jargon unless it is immediately explained.

## Avoid

- Terse bullet-only answers unless the user explicitly asks for brevity.
- Dense shorthand like "Issue: X. Cause: Y. Fix: Z." with no connective explanation.
- Overuse of nested bullets.
- Abrupt transitions.
- Assuming the reader already understands the architecture.

## Preferred Response Shape

1. Direct answer in 1-2 sentences.
2. Short sections with clear Markdown headers.
3. Brief paragraphs that explain mechanism and reasoning.
4. Bullets only where list structure genuinely helps.
5. If relevant, include concrete remediation steps and prevention tips.

## Commit Style
- Prefer Conventional Commit style with chapter scope (for example, `feat(chapter-03): ...`).
- Keep commit messages specific to operational or learning outcomes.

## Agent Workflow
- Before substantial changes, review the latest 3 commits for context and style alignment.
- Preserve and extend Kotlin-first notebook usability in JVM-based workflows.
- If runtime/config behaviour changes, update both implementation and related docs.
- In final handoff, always report:
  - what changed
  - why it changed
  - how to validate it
